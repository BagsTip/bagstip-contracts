# BagsTip — contract decisions v1

Owner: contracts. Share with API (Midhan) + frontend. Revise with PRs.

---

## Decisions

1. **Tx builder:** Client wallet — user signs. API is called **after** on-chain confirmation (e.g. log tip / update DB).

2. **IDL:** Generated by Anchor as `bagstip_tipvault.json` (from crate `bagstip_tipvault`). Client uses `@coral-xyz/anchor` **Program** (or equivalent) with that IDL. If you rename the program, the filename follows the program name.

3. **Release signer:** **Creator wallet only** — enforced on-chain. **No** backend co-sign on the release instruction.

4. **PDA seeds (v1 proposal):** `["escrow", creator_handle_bytes]` with handle normalized (e.g. UTF-8 lowercased). **One vault PDA per creator handle** — see *Open design note* below.

5. **Program ID:** `873HALYp7gZx6tDzPFoxa5UrA7uRJ2eafmCbqwHxSKFH` (devnet). Config: single env var (e.g. `BAGSTIP_PROGRAM_ID`).

6. **Token:** **SOL only** for MVP. Amount in **lamports (`u64`)**. No SPL.

7. **Release gate (product + chain):**
   - **Off-chain:** API only allows “start release flow” when **verified = true** (and any other product rules).
   - **On-chain:** Instruction still only checks **creator signature** + program rules (e.g. tip state, recipient). The program **does not** read the HTTP API; “verified” is **enforced by the app/API before** the user builds the tx. If you later need on-chain proof of verification, that is a **v2** change (e.g. verifier signature or PDA).

---

## Open design note — one PDA per handle

A **single escrow PDA per handle** means **one vault balance per creator** (tips aggregate). That matches a simple “creator tip jar” model.

If the product needs **one on-chain record per tip** (separate pending rows, separate release per tip), use **additional seeds** (e.g. `tip_id` or hash) so each tip gets its own PDA, or store multiple tips in a **larger account** with a capped list (watch account size / rent).

Confirm with product which model MVP uses before locking `release` semantics.
